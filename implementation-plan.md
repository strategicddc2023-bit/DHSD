# แผนพัฒนาระบบ district_health_system_ddc (ฐานงาน)

เอกสารนี้ใช้เป็นฐานในการทำงานจริงของโปรเจกต์ โดยแยกงานเป็น Phase ชัดเจน ครอบคลุม React + GitHub + Vercel + Supabase และยังไม่ลงรายละเอียดโค้ด

## ขอบเขตระบบ
- ระบบรายงานผลข้อมูล พชอ. แบบศูนย์กลาง
- รองรับข้อมูลปีงบประมาณ 2566-2570
- รองรับการกรอกข้อมูล 4 กลุ่ม (Input), การประมวลผล KPI (Process), และ Dashboard/Export (Output)
- เป้าหมายเร่งด่วนระยะแรก: ต้องมีหน้า Dashboard ที่แสดงแผนที่ครอบคลุม สคร. 13 เขต
- ทิศทางหน้าเว็บระยะแรก: พัฒนาเป็น Single-Page Application (SPA) ในสไตล์ Landing Page เพื่อให้เห็นภาพรวมการใช้งานชัดเจน

## โครงสร้างเทคโนโลยี
- Frontend: React (แนะนำ Next.js)
- Source Control: GitHub
- Deploy: Vercel
- Backend/Data/Auth/Storage: Supabase

## Phase 0: Discovery และสรุปข้อกำหนด
วัตถุประสงค์: ล็อกความเข้าใจร่วมกันก่อนเริ่มพัฒนา

สิ่งที่ต้องทำ
- ยืนยันสูตร KPI 3 ตัวชี้วัด และเกณฑ์ Scoring
- ยืนยันสิทธิ์ผู้ใช้ (ส่วนกลาง, สคร./สปคม., จังหวัด/อำเภอ)
- สรุป Data Dictionary กลาง (field, type, required, validation)
- สรุปรูปแบบรายงานและไฟล์ส่งออกที่ต้องใช้จริง (Excel/PDF)

ผลลัพธ์
- เอกสาร Requirements ฉบับยืนยัน
- เอกสาร Data Dictionary
- เอกสาร User Role Matrix

## Phase 1: Project Setup และโครงสร้างพื้นฐาน
วัตถุประสงค์: ทำให้ระบบพร้อมพัฒนาแบบทีม

สิ่งที่ต้องทำ
- สร้าง GitHub Repository และกำหนด Branch Strategy
- สร้าง React/Next.js Project พร้อมโครงสร้างโฟลเดอร์มาตรฐาน
- กำหนดมาตรฐานการแยกไฟล์โค้ดให้ชัดเจน (แยกตามหน้า, components, services, types) เพื่อดูแลง่ายและขยายระบบได้
- ตั้งค่า Vercel (Preview/Production)
- สร้าง Supabase Project
- ตั้งค่า Environment Variables ระหว่าง Local, Preview, Production
- ตั้งค่า `.gitignore` ให้ครอบคลุมไฟล์ความลับทั้งหมด เช่น `.env`, `.env.local`, `.env.*`
- กำหนดนโยบายทีมชัดเจนว่า "ห้ามนำไฟล์ .env ขึ้น GitHub" และให้ใช้ Secret/Environment Variables ของ Vercel และ Supabase แทน

ผลลัพธ์
- Repo พร้อมใช้งาน
- Deploy แรกขึ้น Vercel สำเร็จ
- เชื่อม Frontend กับ Supabase ได้
- ความลับระบบไม่รั่วจากการ commit ไฟล์ `.env`

## Phase 2: ออกแบบฐานข้อมูลและความปลอดภัย
วัตถุประสงค์: วาง schema และสิทธิ์ข้อมูลให้ถูกต้องตั้งแต่ต้น

สิ่งที่ต้องทำ
- ออกแบบตาราง Master และ Transaction ตามเอกสาร IPO
- ออกแบบความสัมพันธ์จังหวัด/อำเภอ/เขต/ปีงบประมาณ/รอบรายงาน
- ออกแบบตารางรับข้อมูลระยะแรก (Initial Import Table) สำหรับเก็บเฉพาะหัวตารางที่ต้องใช้ก่อน ดังนี้
  - หน่วยงาน: Dropdown ค่า `สคร.1` ถึง `สคร.13`
  - จังหวัด: เลือกได้เฉพาะจังหวัดในประเทศไทย
  - อำเภอ: เลือกได้เฉพาะอำเภอของจังหวัดที่เลือก (ล็อกการมองเห็นเฉพาะรายการที่เกี่ยวข้อง)
  - ประเด็นโรค/ภัยสุขภาพ: ข้อความให้ผู้ใช้กรอกเอง
- กำหนดว่าหัวข้ออื่นยังไม่เก็บในระยะแรก และจะเพิ่มใน Phase ถัดไป
- ทำ Row Level Security (RLS)
- ตั้ง Supabase Auth และ role mapping
- ตั้ง Storage Bucket สำหรับเอกสารแนบ พร้อม policy

ผลลัพธ์
- Schema พร้อมใช้งานจริง
- สิทธิ์การเข้าถึงข้อมูลใช้งานได้ตามบทบาท
- อัปโหลด/ดาวน์โหลดไฟล์แนบได้อย่างปลอดภัย

## Phase 3: พัฒนาระบบ Input (ฟอร์ม 4 กลุ่ม)
วัตถุประสงค์: ให้หน่วยงานกรอกข้อมูลได้ครบและถูกต้อง

สิ่งที่ต้องทำ
- เริ่มจากฟอร์มระยะแรก (MVP Input) เพื่อรองรับการนำเข้าข้อมูลบางส่วนก่อน โดยมีเฉพาะ
  - หน่วยงาน (Dropdown สคร.1-สคร.13)
  - จังหวัด (ประเทศไทย)
  - อำเภอ (กรองตามจังหวัดที่เลือกเท่านั้น)
  - ประเด็นโรค/ภัยสุขภาพ (ข้อความ)
- ตั้งกติกา dependent dropdown จังหวัด -> อำเภอ เพื่อไม่ให้แสดงอำเภอที่ไม่เกี่ยวข้อง
- เตรียมโครงสร้างฟอร์มให้ขยายหัวข้ออื่นในอนาคตได้โดยไม่กระทบข้อมูลเดิม
- แยกไฟล์การพัฒนาแต่ละหน้าอย่างชัดเจน และทำ routing/navigation ให้ทุกหน้าเชื่อมโยงถึงกันได้
- เน้นพัฒนาหน้าเว็บเพื่อให้เห็นภาพการใช้งานจริงก่อน (UI flow first) โดยยังไม่เปิดใช้การแยกสิทธิ์
- พัฒนาโครงหน้าแบบ Landing Page/SPA โดยมี Hero Section หรือ Hero Banner เป็นจุดเริ่มของประสบการณ์ผู้ใช้
- เพิ่ม Scroll Indicator เพื่อสื่อว่าหน้ายังมีเนื้อหาส่วนถัดไปและช่วยนำทางการเลื่อนหน้า
- สร้างหน้าฟอร์มข้อมูลตั้งต้นพื้นที่และเวลา
- สร้างหน้าฟอร์มโครงการ/การวิเคราะห์ปัญหา/5 Clusters
- สร้างหน้าฟอร์มติดตามประเมินผล (KPI/Target/ผลลัพธ์/อุปสรรค)
- สร้างหน้าสรุป/ข้อเสนอแนะ/รางวัล/แนบเอกสาร
- เพิ่ม Validation, Draft/Submit workflow

ผลลัพธ์
- ผู้ใช้กรอกและบันทึกข้อมูลครบ 4 กลุ่มได้
- ลดข้อผิดพลาดจากข้อมูลไม่ครบ/รูปแบบผิด

## Phase 4: พัฒนาระบบ Process (Calculation + Integration)
วัตถุประสงค์: คำนวณและสรุปผลอัตโนมัติ

สิ่งที่ต้องทำ
- สร้าง Calculation Engine สำหรับ KPI 3 ตัว
- แปลงร้อยละเป็นคะแนนตามเกณฑ์ที่กำหนด
- สรุปยอดรวม (focus district, รางวัล, บรรลุ/ไม่บรรลุ)
- ทำ clustering summary ตาม 5 Clusters
- เพิ่มตารางสรุปผลสำหรับ Dashboard (เช่น summary รายอำเภอ/รายเขต/รายปี) เพื่อให้ดึงข้อมูลซ้ำมาแสดงผลได้เร็ว
- กำหนดรอบการ refresh/recompute ตารางสรุป (ทุกครั้งที่ submit หรือแบบ scheduled job)
- เตรียม data pipeline สำหรับ import Excel/API (P0)

ผลลัพธ์
- ระบบคำนวณผลได้ถูกต้องและตรวจสอบย้อนกลับได้
- มีชุดข้อมูลสรุประดับอำเภอ/เขต/ประเทศ
- มีตารางพร้อมใช้งานสำหรับดึงข้อมูลกลับมาแสดงหน้า Dashboard โดยไม่ต้องคำนวณใหม่ทุกครั้ง

## Phase 5: พัฒนาระบบ Output (Dashboard + รายงาน)
วัตถุประสงค์: แสดงผลเพื่อการตัดสินใจเชิงบริหาร

สิ่งที่ต้องทำ
- พัฒนา Interactive Map ระดับประเทศ/เขต โดยต้องแสดงผลครบ สคร. 13 เขต
- ออกแบบ Immersive Background สำหรับหน้า Dashboard/Landing โดยใช้ภาพกราฟิกที่มีมิติความลึก (Depth of Field) เพื่อเพิ่มความน่าสนใจ
- พัฒนา Dashboard charts (Donut/Radar, Bar, KPI Cards)
- พัฒนาตารางค้นหา/กรองข้อมูลตามปี/พื้นที่/สถานะ
- ดึงข้อมูลจากตารางสรุปที่เตรียมไว้ (pre-aggregated table/materialized view) เป็นแหล่งหลักของหน้า Dashboard
- พัฒนา Export Excel และ PDF

ผลลัพธ์
- ผู้บริหารเห็นภาพรวมและเจาะลึกข้อมูลได้ทันที
- ส่งออกข้อมูลประชุมได้จากระบบโดยตรง

## Phase 8: เตรียมรองรับ SSO กรมควบคุมโรค
วัตถุประสงค์: ออกแบบระบบให้พร้อมเชื่อม SSO ในอนาคต โดยไม่กระทบโครงสร้างที่พัฒนาไปแล้ว

สิ่งที่ต้องทำ
- ออกแบบ authentication architecture ให้รองรับ SSO ของกรมควบคุมโรค
- แยกชั้น auth/client/service ไว้ล่วงหน้า เพื่อสลับจาก login ชั่วคราวไป SSO ได้ง่าย
- กำหนดจุดเชื่อมต่อ (callback URL, token handling, session lifecycle) ในระดับสถาปัตยกรรม
- ระยะแรกยังไม่แยกสิทธิ์การใช้งาน (Authorization) เพื่อโฟกัสการเห็นภาพรวมของหน้าเว็บก่อน
- วางแผนรองรับ ThaiD/OIDC โดยแยก 2 ชั้นให้ชัดเจน
  - Authentication: ThaiD/SSO ยืนยันตัวตนว่าเป็นใคร
  - Authorization: ระบบฐานข้อมูลของเราเป็นผู้ตัดสินว่าเข้าใช้งานได้หรือไม่
- ออกแบบตารางสิทธิ์ผู้ใช้งาน (เช่น `app_users`) เพื่อทำ allowlist และสถานะการอนุมัติ
  - ฟิลด์หลักที่ต้องมี: `thai_d_sub` (หรือรหัสอ้างอิงจาก SSO), `role`, `agency_code`, `province_code`, `district_code`, `status`, `last_login_at`
  - สถานะเริ่มต้นแนะนำ: `pending`, `active`, `suspended`
- ออกแบบ Login Decision Flow
  - ผู้ใช้ login ผ่าน ThaiD สำเร็จ -> ระบบอ่าน identifier จาก token claim
  - ตรวจใน `app_users`
  - ไม่พบรายชื่อ หรือสถานะไม่ใช่ `active` -> ปฏิเสธการเข้าใช้งาน (หน้าไม่มีสิทธิ์)
  - พบและ `active` -> อนุญาตเข้าใช้งาน และแนบ role ลง session
- เตรียมกระบวนการอนุมัติสิทธิ์ผู้ใช้ใหม่ (approval workflow) โดย Super admin/Admin
- จัดเก็บ audit log สำหรับเหตุการณ์ login success/denied เพื่อความปลอดภัยและตรวจสอบย้อนหลัง

ผลลัพธ์
- โครงสร้างระบบพร้อมต่อยอด SSO ได้เมื่อได้รับรายละเอียดจริง
- ลดความเสี่ยงการรื้อโค้ดใหญ่เมื่อเริ่มเปิดใช้ SSO จริง
- มีโมเดลตรวจสอบสิทธิ์ที่ชัดเจนว่า "มี ThaiD ไม่ได้แปลว่าเข้าใช้งานระบบได้ทันที"

## สรุปแผนพัฒนา (ล็อกตามที่ตกลงล่าสุด)
### ทำทันที (งานหลักรอบปัจจุบัน)
- พัฒนาเว็บแบบ SPA/Landing Page พร้อม Hero Section/Hero Banner, Immersive Background และ Scroll Indicator
- พัฒนาหน้า Dashboard แผนที่ให้รองรับการแสดงผลครบ สคร. 13 เขต
- เริ่มเก็บข้อมูลเฉพาะหัวตารางระยะแรก: หน่วยงาน (สคร.1-13), จังหวัด, อำเภอ (กรองตามจังหวัด), ประเด็นโรค/ภัยสุขภาพ
- แยกไฟล์โค้ดเป็นสัดส่วนชัดเจนรายหน้า และเชื่อม navigation/routing ระหว่างหน้าให้ครบ
- เพิ่มตารางสรุปผลสำหรับ Dashboard เพื่อดึงข้อมูลกลับมาแสดงผลได้เร็ว (ไม่คำนวณใหม่ทุกครั้ง)
- ควบคุมความปลอดภัยความลับระบบ: ห้ามนำไฟล์ `.env` ขึ้น GitHub และใช้ Environment Variables/Secrets เท่านั้น
- เริ่ม Phase 4: วางโครง Calculation Engine (KPI input, KPI results, scoring function, และ summary view)
- นโยบายข้อมูลระยะนี้: สร้างเฉพาะโครงสร้างตาราง/ฟังก์ชัน/วิวตามแผนก่อน และยังไม่ seed ข้อมูลตัวอย่างเข้าระบบ

### สถานะการดำเนินงานตาม Phase (อัปเดตล่าสุด)
- `Phase 0` เสร็จ: Discovery, Requirements baseline, Data Dictionary baseline, Role Matrix baseline
- `Phase 1` เสร็จ: Setup React/Next.js, โครงสร้างโฟลเดอร์, build pipeline, นโยบาย `.env` ไม่ขึ้น GitHub
- `Phase 2` เสร็จเชิงโครงสร้าง: ออกแบบฐานข้อมูลหลัก, master agencies/provinces/districts, mapping สคร.-จังหวัด, RLS MVP
- `Phase 3` เสร็จ MVP: ฟอร์ม 4 ฟิลด์, จังหวัดกรองตามหน่วยงาน, อำเภอกรองตามจังหวัด, บันทึกลงฐานข้อมูลจริง
- `Phase 4` เริ่มแล้ว: วางโครง KPI engine ในฐานข้อมูล (ตาราง + ฟังก์ชัน + trigger + summary view) โดยยังไม่ใส่ข้อมูลตัวอย่าง
- `Phase 5` กำลังทำ: Dashboard preview, summary cards, filters, ตารางข้อมูลล่าสุด, coverage ตาม สคร.
- `Phase 6-8` ยังไม่เริ่มบังคับใช้จริง: Alert/Forecast, UAT hardening, SSO rollout

### วางโครงไว้ก่อน (ยังไม่ใช่งานหลักรอบนี้)
- วางโมเดลสิทธิ์ผู้ใช้ `super_admin`, `admin`, `user` ในฐานข้อมูลและสถาปัตยกรรม แต่ยังไม่บังคับใช้งานจริงทุกหน้า
- เตรียมรองรับ SSO/ThaiD (OIDC) ในระดับสถาปัตยกรรมและ data model
- ใช้หลักการ "Authentication แยกจาก Authorization" โดยให้ฐานข้อมูลระบบเป็นตัวตัดสินสิทธิ์เข้าใช้งานจริง
- เตรียม allowlist (`app_users`), สถานะผู้ใช้ (`pending/active/suspended`), และ audit log สำหรับเปิดใช้ในเฟสถัดไป
